[개요]
- 실력 향상은 뭐다? 피드백
- 하지만 양질의 피드백은 항상 받을 수 있다? 없다!
- 그래서 오늘도 책을 끄적이며 혼자 푼 내용을 셀프 피드백 한다!
[문제]
https://leetcode.com/problems/tree-node/description/
```
# Write your MySQL query statement below
select 
t.id
, case 
when t.p_id is null then 'Root'
when t.p_id is not null and exists (
	select 1
    from Tree tt
    where tt.p_id = t.id
) then 'Inner'
when t.p_id is not null and not exists (
    select 1
    from Tree tt
    where tt.p_id = t.id
) then 'Leaf'
 end as `type`
from tree t;
```
- 기존의 쿼리가 중복이 많이된다
    - t.pd is not null > 중복
    - not exists 대신 else 활용 가능
[해결]
  ```
  select 
  id
  , case
        when p_id is null then 'Root'
        else
            case
                when exists (
                    select 1
                    from Tree tt
                    where tt.p_id = t.id
                ) then 'Inner'
                else  'Leaf'
            end
    end as `type`
  from tree t;
  ```
[문제]
- 위 쿼리를 다른 사람들은 많이들 in을 써서 사용한다고 한다
- 내가 겪은 문제는 아니지만 미래에 문제가 될 수 있는 것도 역시나 짚고간다!
- exists vs in (성능 관점)
- in은 in 서브쿼리를 최적화해서 세미 조인 형태로 변환하거나 내부적으로 distinct 처리를 하려고 함. 서브쿼리의 양이 방대해지면 임시 테이블 생성 등의 비용이 발생
- exists는 존재 여부만 확인하면 즉시 스캔 중단함. 대용량 데이터에서 성능상 유리.
[해결]
- 사용 기준을 세워서 쓰면 된다고 생각
- EXISTS를 써야 할 때: 바깥쪽(메인) 테이블이 작고, 안쪽(서브쿼리) 테이블이 클 때
- IN을 써야 할 때: 바깥쪽(메인) 테이블이 크고, 안쪽(서브쿼리) 테이블이 작을 때
