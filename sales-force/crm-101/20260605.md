Q1. Enhanced for문(for-each)의 실체: 자바에서 for (String s : list)를 쓰면 컴파일러는 내부적으로 이를 어떤 코드로 변환할까요? 일반 배열(Array)을 돌 때와 ArrayList 같은 컬렉션을 돌 때 내부 변환 방식이 왜 다를까요?
> 컴파일러는 해당 배열의 index를 0번부터 끝까지 읽는 코드로 변환합니다. ArrayList도 내부적으로는 배열을 사용하지만 Iterator(반복자) 객체를 사용하는 코드로 변환합니다. 
```
List<String> list = Arrays.asList("A", "B", "C");
Iterator var2 = list.iterator();

while (var2.hasNext()) {
    String s = (String)var2.next();
    System.out.println(s);
}
```
>> 다형성과 구조적 효율성을 위해서 이러한 변환을 시도합니다. 자바의 for-each는 ArrayList 뿐만 아니라 LinkedList, HashSet, TreeSet 등 Iterable 인터페이스를 구현한 모든 객체에서 사용 가능한데, 이떄마다 매번
get(i)로 인덱스로 접근은 어렵다. 연결리스트는 매 처음부터 찾아가면 시간복잡도가 최악이고, HashSet은 인덱스 자체가 없는 자료구조라 get(i) 메서드 자체가 없습니다. 
따라서 자바는 다음 값이 있으면 가져오게 표준화된 방법인 iterator 구조로 통합해버린 것이다. 
>> 반면 일반 배열은 자바 컬렉션 프레임워크에 속하지 않는 로우 레벨 구조이다. Iterator를 만들어내는 기능이 없다. 대신 메모리가 연속적으로 붙어있으니 인덱스 접근이 빨라, 컴파일러가 굳이 무거운 객체를 쓰지 않는다.단순 인덱스 카운터 반목문으로 최적화해준다.
