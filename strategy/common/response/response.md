[고민 포인트]
- 클라이언트 관점
클라이언트에서 넘겨줘야하는 정보가 무엇인지, 바디가 정말 필요한지, 상태 코드만으로 괜찮은지
어떻게 하면 프론트엔드 개발자가 내 API를 한 번에 이해하고 예외 처리를 편하게 할까?
> 

- 성공과 실패 케이스 각각 어떤 정보가 필요한지
운영과 개발 환경 에러 상세 다르게 가져갈건지

일관성
- 팀 단위로 리턴 방식을 통일할 필요가 있는지

성능
- 페이로드 크기 최소화할 필요 있는지
예) 불필요한 필드(예: 내부 DB ID) 제외, JSON만 사용.
캐싱 헤더(Etag, Cache-Control), 버저닝(/v1/) 고려.

보안
- 민감 정보(비밀번호 등) 절대 응답에 넣지 않는다 등.


[트레이드 오프]

> 조금 더 유연한 개발이 가능하다
- 커스텀 리스폰스 엔티티 활용
> SuccessResponse, ErrorResponse 
> 리턴 방식의 규격을 통일하고, 강제할 수 있다
