[판단의 기준]
에러 로그의 판단 기준은, 누가 이 로그를 보고 무엇을 할 수 있는가? 룰 결정해야한다.
로그의 목적이 개발자가 봄며, **'단순 알림'**인지, **'디버깅'**인지,
아니면 사용자가 보며, **'사용자 피드백'**인지에 따라 결정.

[예외 유형별 차등]
- 비즈니스 예외 (예: 중복 계정)
  WARN 또는 INFO
  에러 코드, 메시지
- 시스템/상태 예외 (예: 상품 없음)	
  ERROR
  에러 코드, Stack Trace
- 인프라/런타임 예외 (예: DB Down)
  ERROR (Critical)
  전체 정보, 원인, 환경 변수

[판단을 남기기 위해 활용할 수 있는 것]
Throwable 클래스를 상속받는다고 가정
1. Throwable 메서드별 활용 가치
   getMessage()
   필수. 예외 발생 시점의 구체적인 상황을 설명합니다.	대부분 이 정보가 가장 중요합니다.
   getStackTrace()
   디버깅의 핵심. 에러가 발생한 정확한 코드 라인을 추적합니다.	로그 프레임워크(Logback, Log4j2)가 자동으로 처리하도록 맡기는 것이 좋습니다.
   getCause()
   연쇄 예외(Chained Exception) 확인 시 사용.	NullPointerException보다는 SQLException처럼 근본 원인이 따로 있을 때 유용합니다.
   getLocalizedMessage()
   다국어 처리가 필요한 클라이언트 응답용. 서버 로그용으로는 getMessage()와 동일한 경우가 많아 굳이 쓰지 않습니다.

2. 로깅 시 고려해야 할 3가지 포인트
   AOP에서 AfterThrowing 등을 사용할 때 아래 기준을 적용해 보세요.
   Context(문맥) 포함 여부: 단순히 에러 메시지만 찍는 게 아니라,
   어떤 요청 파라미터가 들어왔을 때 에러가 났는지(Method Arguments)를 함께 남기는 것이 가장 유의미합니다.
   AOP의 JoinPoint를 통해 매개변수를 로그에 담으세요.
