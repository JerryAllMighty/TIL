- G26: 정확하라
>> 결정 시에 정확하게 하라는 뜻인데, 예를 들면 호출 함수의 null 여부
조회 결과가 하나뿐인지 확실하게 돈을 다룬다면 정수를 사용, 반올림 올바르게 처리
병행 특성으로 인해 동시에 갱신할 가능성이 있다면 적절한 잠금 매커니즘 구현

나는 이 중 null 체크를 더 하려고 했다.
```
public ResponseEntity<> getProducts(@PathVariable("id") long id) {
return ResponseEntity.ofNullable(productService.findProduct(id));
}
```

여기서 id 값을 받는데, 이걸 long으로 할지 Long으로 할지 고민해보자.
null 일 수가 있나? 그래서 체크를 해야하나? 를 고민했고.
결론은 id가 비면 404, 넘버 포맷이 안 맞다면 400으로 리턴되니 해당 부분을
null 체크를 위한 래핑 클래스를 과하다고 판단. 원시 타입을 적용했다.


하나 더.
```
@Override
public Product findProduct(long id) {
return productRepository.findById(id)
.orElseThrow();
}
```
이렇게 있다면 조회 값이 null 이더라도 예외를 throw 할 수 있다는 점. 


** 기존에도 원래 썼던 요지지만 포인트는 책을 보고 고민 포인트를 잡아서 짚고 넘어갔다는 점이 아닌가 싶다.
