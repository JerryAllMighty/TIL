---------------------
프로젝트 구조부터 고민이다.
포트폴리오로 사용할 프로젝트이고, 우선은 MVP를 생각하고 있다.
하지만 어떤 구조를 잡아야하고, 왜 이런 구조를 잡아야하는지 한 번 짚고 가야겠다 싶어서 그 고민 과정을 정리해본다.
우선 내 지금 상황이 어떠한가 문제를 정의해보았다.


[고려 포인트]
프로젝트는 MVP로 우선 만들 예정이라서
- 낮은 러닝 커브
- 명확한 책임 소재 분리
- 빠른 개발과 배포가 필요
- 비즈니스 로직이 복잡한가?
- 우선 나는 1인 개발이라 팀 단위로 개발하는 것에 있어서의 고민 포인트는 제외하였다.



[배포]
- 빠른 배포를 할거면 단일 프로세스로 배포하는 게 좋다.
- 모놀리스는 통합 코드베이스/ 단일 배포라 빠름
- 비교해보고 모듈형 모놀리스도 고려해볼 수 있다
- 멀티 모듈 이상으로 가면 각 단위별로 배포가 이루어질텐데,
  애플리케이션이 커져서 이전에 백오피스, 점주, 배치 등 각 단위별 변경 사항이 나뉘어서 나머지에 영향도를 줄이는게 좋을 정도면 그 때부터 좋다고 생각.
  배포 파이프라인에 정적 분석이나 다른 자동화 테스트도 넣는다면 더더욱 분리해야함.
  하지만 이번 프로젝트는 그 정도로 커질 것 같진 않다.
>> 빠른 기능 개발, MVP가 목표라 배포는 단일 프로세스가 좋겠다 판단했다.



[관심사 분리]
- 모놀리스는 규모가 커지면 모듈간 경계가 무너져 스파게티 될 가능성 높다
- 멀티 모듈의 경우는 관심사 분리가 명확하다. 또한 의존성 관리가 명확해 코드 레벨에서 의존 방향 강제 가능하다.

>> 그리고 개인적으로 나는 멀티모듈 구조였을 때 개발이 좀 진행되고 나면
서비스 레이어에서 다른 레이어 의존이 너무 많아져서 나중에 유지보수할 때 부담이 많아졌음.
또 변경하려면 너무 많은 회귀 테스트를 해야하는 경우도 생김
따라서 가급적이면 모듈은 작게 가져가고 싶었다. 근데 정산이나 배치까지는 직접 해 볼 생각이 있었기 때문에
멀티 모듈은 불가피한 상황이라고 판단했다. 배치는 독립적인 기능 단위가 맞다고 생각해 모듈 따로 분리해야겠다고 생각했다.
> 일단 모듈 분리 기준은 기능 기준으로...!
이렇게 생각하니 상품 데이터 등도 따로 넣어줘야 사용자가 사용할 수 있으니 admin도 같이 분리를 해서 시작하자...! 라고 결론
>> 그럼 이제 이 안에서 모듈들 안에서 어떻게 구조를 또 나누어서 가지고 갈까 고민해야한다.
각 모듈은 레이어별로 나눌지, 아니면 도메인별로 다음으 나눌지 고민했다.

> 각기 다른 목적이 있기에 비교해보고 전략적으로 아키텍쳐를 가져가기로 했다.
Layered vs Hexagonal vs DDD

[Layered]
- 단방향 의존성의 레이어드 아키텍쳐를 활용하면 코드 가독성, 유지보수성 높일 수 있겠다 싶었다
>> 일단 의존성 관리 용이하고, 테스트도 수월한 건 이점이다. 기준 정보가 되는 상품 데이터를 등록하는 등의 용도가
많을 것 같은 admin은 레이어드로 가독성과 테스트 용이, 디버깅이 쉬운 레이어드로 가는 것으로 선택했다.
>> 주문과 결제쪽은 핵심으로 생각하는 기능들이라 MVP에서 시작해서 다른 기능들을 쭉쭉 추가해볼 생각이고,
따라서 변화에 유연해야한다고 판단했다. 기존의 레이어드 아키텍쳐는 하나의 서비스 레이어가 아주 비대해지는? 일이 있어
테스트가 어렵고 의존성이 높아 특히 상품 등의 도메인이 변경되면 엮인 부분의 테스트를 아주 많이 해야했다.

[DDD]
- DDD가 복잡한 비즈니스 로직 관리 용이
- 비즈니스 변화에 유연하게 대응, 유지 보수성 높음
>> 주문과 결제 쪽은 이렇게 도메인 별로 묶는 방식을 가져가서 한 엔티티의 수정이 다른 기능에 영향을 주는 부분을 최소화해야한다고 생각했다.

[Hexagonal]
- 만약 db를 바꿔야하는 상황이 있을 수 있다면 헥사고날 고려할 수 있다.
>> 실제 RDB 변경될 부분은 해당 사항 없을 것 같다.
>> 배치 쪽에 적용하기로 했는데 그 이유는....다음 기회에...!
