[언제 쓰면 좋을까?]
- 필요한 원소를 컴파일 타임에 다 알 수 있는 상수 집합이라면 항상 열거 타입을 사용하자
- 하나의 메서드가 상수별로 다르게 동작해야할 때는 상수별 메서드 구현을 사용하자
- 열거 타입 상수 일부가 같은 동작을 공유한다면 전략 열거 타입 패턴 고려해보자


[근본 사용 이유]
- 열거 패턴의 단점 때문이다. 
- 잘못된 상수 비교해도 컴파일 시에 체크도 못한다.
- 평범한 상수를 나열한 것뿐이라서 컴파일시 그 값이 클라이언트 파일에 그대로 새겨진다.
- 또한 상수 값이 바뀌면 클라이언트도 다시 컴파일해야한다.
- 디버깅도 숫자로만 보여서 의미가 없다.

[설계 포인트]
- 상태와 행위의 응집도
로직이 데이터와 반드시 같이 움직여햐하는지? 그렇다면 Enum 안에서 캡슐화를 활용 가능.
- 상수별 메서드 구현
로직이 복잡하고 상수마다 구현이 다르면 abstract 사용해서 각 상수에서 오버라이딩 가능
- 타입 안정성
잘못된 상수 비교 방지도 할 수 있지만, 다른 열거 타입을 강제함으로써 코드 전체의 가독성과 안정성을 높일 수 있다.
- 불변으로 설계의 이점
기본적으로 public static final. 이는 스레드 세이프하다는 것을 의미. 공유 자원이어도 상태가 변하지 않아서 안심하고 사용 가능.
- 예외처리
역직렬화시 Enum 타입 아니면 에러날 수 있으니 예외처리 필요.
- 외부 의존성 배제
Enum은 순수하게 계산이나 판단 로직으로만 사용. DB 저장이나 외부 API 호출 등은 서비스 레이어에서.
- 상태에 따라 행위가 그룹화되나?
중첩 Enum 전략, 전략 패턴 활용 고려

[문제점]
- 상속 불가
java.lang.enum을 상속 받고 있다. 다른 클래스 상속이 안된다.
- Ordinal 사용 지양.
만약에 상수 추가된다면 기존의 로직 깨질 수도. 항상 String 사용 권장
- 역직렬화
JSON 데이터를 Enum으로 역직렬화시 존재하지 않는 값이 들어오면 예외가 발생.
