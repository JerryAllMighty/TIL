# [판단의 기준]
예외나 에러의 삼각도를 먼저 판단한다. 이후 누가 이 로그를 보고 무엇을 할 수 있는가?를 결정해야한다.
로그의 목적이 개발자가 보며, **'단순 알림'**인지, **'디버깅'**인지,
아니면 사용자가 보며, **'사용자 피드백'**인지 결정한다.

# [예외 유형별 차등]
- 비즈니스 예외 (예: 중복 계정)   
  WARN 또는 INFO
  에러 코드, 메시지 등을 남기면 좋다.
  getMessage() : 필수. 예외 발생 시점의 구체적인 상황을 설명
  getLocalizedMessage() : 다국어 처리가 필요한 클라이언트 응답용. 서버 로그용으로는 getMessage()와 동일한 경우가 많아 굳이 쓰지 않음.

- 시스템/상태 예외 (예: 상품 없음)   	
  ERROR
  에러 코드, 메시지, Stack Trace 등을 남기면 좋다.
  getStackTrace() : 디버깅의 핵심. 에러가 발생한 정확한 코드 라인을 추적.
  로그 프레임워크(Logback, Log4j2)가 자동으로 처리하도록 맡기는 것이 좋음.

- 인프라/런타임 예외 (예: DB Down)   
  ERROR (Critical)
  전체 정보, 원인, 환경 변수 등을 남기면 좋다.
  로그 프레임워크(Logback, Log4j2) 활용 stack trace 등을 남기면 좋다.
  getCause()
  연쇄 예외(Chained Exception) 확인 시 사용. NullPointerException보다는 SQLException처럼 근본 원인이 따로 있을 때 유용

어떤 요청 파라미터가 들어왔을 때 에러가 났는지(Method Arguments)를 함께 남기려면 어떻게 할지?
   AOP에서 AfterThrowing 등을 사용할 때 아래 기준을 적용해 보세요.
   Context(문맥) 포함 여부: 단순히 에러 메시지만 찍는 게 아니라,
   는 것이 가장 유의미합니다.
   AOP의 JoinPoint를 통해 매개변수를 로그에 담으세요.


[결론]
- 클라이언트에게 보여주는 부분인지, 개발자가 서버에서 일어날 일을 보는 목적인지 정한다
- 해당 예외의 심각도를 먼저 결정한다. warn, info, debug, error 등등. 
- 해당 예외의 심각도에 따라 클라이언트와 서버에서 어떤 역할을 할지 정하면 된다. 
- 비즈니스 예외라 클라이언트에 정보만 제공해주면 된다면 메세지나 에러 코드 정도만 넘겨주면 된다. Throwable 클래스의 메서드를 활용 가능하다.
- 하지만 상황에 따라 디버깅 등이나 상세 정보가 필요하다면 로그 프레임워크를 활용할 수 있다.
- 전체 맥락이나 요청 파라미터도 남겨야한다면 AOP joinPoint를 고민해 볼 수 있다.
