<details>
<summary> 외부 API와의 연동 시에 왜 원시 타입보다는 래퍼 클래스를 사용하는 것이 더 안전할까요? </summary>
<div markdown="1">
보통 외부 기관 등에 의해서 동작하기에 개발자가 제어할 수 없는 영역이라 값을 보장할 수 없기에 래퍼 클래스가 더 유리합니다.
또한 원시 타입은 기본형을 리턴하기에 명확한 원인 파악이 어려울 수 있습니다. 실제 값이 0인지, 전송이 원할하지 않아 실패한 값인지 판단이 어렵습니다. 하지만 null 체크를 통하면 명확하게 의도된 예외처리를 할 수 있는 부분도 판단의 근거가 됩니다.
</div>
</details>


<details>
<summary> Java에서 null 값을 안전하게 처리하기 위해 주로 사용되는 방법은 어떤 것들이 있나요? </summary>
<div markdown="1">
Optional 객체를 활용할 수 있습니다. Null을 포함할 수 있는 개체로 자바 8 이후부터 나온 기능입니다. 혹은 방어적인 프로그래밍을 사용하여 메서드 도입부에 파라미터 등이 null인지 체크하는 방법도 있습니다. 
혹은 @nonNull 같은 어노테이션을 활용하여 컴파일이나 런타임에 체크하는 방법도 있습니다
</div>
</details>

<details>
<summary> Optional 클래스를 사용할 때 주의해야할 점은 무엇인가요? </summary>
<div markdown="1">
반환타입으로 설계된 클래스입니다. 따라서 클래스 필드나 메서드 파라미터로 사용하지 않아야합니다. 나주엥 직렬화 등에서 문제가 생길 수 있고, 가독성도 저해합니다.
가능하다면 OptionInt같은 원시 타입을 사용하는 것이 좋습니다. Optional<Integer>는 박싱이 일어나고 나중에 값을 꺼낼 때는 다시 언박싱이 필요해 래퍼 클래스의 생성이 많아진다면 금방 오버헤드를 일으킬 가능성이 있습니다. 
ifPresent().get()같은 Optional 사용의 기본 의미를 퇴색시크는 사용은 지양해야합니다. if not null 체크하던 기존의 로직과 의미상 동일하기 때문입니다. orElse()나 orElseGet() 같은 선언적 메서드 사용을 지향해야합니다. 
</div>
</details>
