Q1. [타입 프로모션 & 바이트코드]
byte a = 10; byte b = 20; 일 때, byte c = a + b; 라고 적으면 컴파일 에러가 발생합니다.
왜 컴파일러는 이를 막을까요? 그리고 실제 JVM 내부(바이트코드 수준)에서는 byte나 short 같은 작은 정수형의 산술 연산이 어떻게 처리되나요?
> 바이트끼리의 연산은 오버플로 가능성으로 인하여 자바 설계자들이 기본 연산 단위를 int로 통일하도록 JLS (자바 언어 명세)를 정했습니다. 그래서 int처럼 큰 단위의 연산 결과를 byte라는 작은 결과에 담으려고 하니 데이터 손실 가능성으로 인해 컴파일 단계에서 에러가 나는 것입니다.
> JVM의 피연산 스택은 기본적으로 4바이트 단위로 작동합니다. iload (Integer Load) 명령어를 이용하여 이 값을 int로 형변환 시켜 스택에 쌓습니다. 이후 int 끼리 산술 연산을 합니다.
> 명시적으로 바이트로 바꾸려하면 강제로 1바이트 크기로 잘라내게 되는 과정을 거칩니다.

Q1-2.
```
byte a = 10;
byte b = 20;

// 1번 케이스 (컴파일 에러 발생)
a = a + b; 

// 2번 케이스 (컴파일 에러가 발생하지 않음!)
a += b;
```
> 1번 케이스와는 다르게 2번 케이스는 컴파일 에러가 나지 않고 정상 작동합니다.  이유가 뭘까요?
>> 대입 연산은 자바 컴파일러가 한 세트로 형변환까지 함께 해주기 때문에 그렇습니다. 대입 연산자는 형변환까지 포함되어 있는 과정이라 그렇습니다.
```
byte b = -1; // 이진수로 1111 1111 (2의 보수 표현)
int i = b;   // byte가 int로 자동 프로모션 됨
System.out.println(i);
```



> 위 코드에서 i를 출력하면 어떤 값이 나올까요? (-1 vs 255 vs 다른 값)
> byte 타입의 -1(1111 1111)이 4바이트인 int 타입으로 늘어날 때, 비트 상에서는 어떤 변화가 일어날까요? 컴퓨터가 부호를 유지하기 위해 채워 넣는 이 현상의 이름과 원리를 아시는 대로 설명해 주세요!

Q3. [부동 소수점의 한계와 실무]
System.out.println(0.1 + 0.2 == 0.3);을 실행하면 false가 출력됩니다. 컴퓨터가 실수를 표현하는 방식(IEEE 754)의 한계 때문인데, 이 현상이 발생하는 정확한 이유와 금융권 같이 정확한 돈 계산이 필요한 실무에서는 이를 해결하기 위해 어떤 클래스를 사용해야 하나요? (그 클래스를 쓸 때 주의해야 할 점도 함께 알고 계신가요?)
